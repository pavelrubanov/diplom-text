\section{Описание решения}
\label{sec:solution}

\subsection{Требования к системе}

Требования сформулированы совместно с консультантом (м.н.с.\ лаборатории Гербарий ГБС РАН, Шкурко~А.~В.)
на основе технического задания.

\subsubsection*{Требования заказчика}

\begin{enumerate}
    \item Автоматическое распознавание формы листа и построение полигональной маски контура.
    \item Определение длины и ширины полигона.
    \item Расстановка точек вдоль перпендикуляров к главной оси с заданным шагом (например, $\frac{1}{2}$ или $\frac{1}{4}$ длины листа).
    \item Просмотр исходного изображения и маски рядом на экране.
    \item Возможность задать масштабный коэффициент ($1~\text{пиксель} = \ldots~\text{мм}$) для вывода измерений в физических единицах.
    \item Экспорт результатов в табличный формат (CSV / Excel).
    \item Пакетная обработка серий изображений.
    \item Возможность ручной корректировки автоматически построенной маски.
    \item Полностью автоматический режим обработки изображения: построение масок всех объектов без участия пользователя, автоматический отбор корректных листьев и запуск измерений.
\end{enumerate}

\subsubsection*{Функциональные требования}

На основе технического задания выделены следующие варианты использования:

\begin{itemize}
    \item \textbf{UC1.} Загрузить набор изображений для обработки.
    \item \textbf{UC2.} Получить маску сегментации по интерактивным подсказкам (точки, ограничивающий прямоугольник).
    \item \textbf{UC3.} Скорректировать маску вручную в режиме кисти/ластика.
    \item \textbf{UC4.} Добавить подтверждённую маску в буфер или удалить её.
    \item \textbf{UC5.} Задать масштабный коэффициент пиксель~→~мм.
    \item \textbf{UC6.} Запустить пакетное измерение всех сохранённых масок.
    \item \textbf{UC7.} Экспортировать таблицу результатов в~CSV~/~XLSX.
    \item \textbf{UC8.} Выбрать тип листа для автоматической обработки (из списка типов, для которых обучена модель отбора).
    \item \textbf{UC9.} Запустить полностью автоматический режим: система строит маски всех объектов на изображении, отбрасывает некорректные с помощью обученного классификатора и добавляет оставшиеся в буфер без участия пользователя.
    \item \textbf{UC10.} Просмотреть результаты автоматического отбора и при необходимости скорректировать отдельные маски вручную (переход в~UC3).
\end{itemize}

\subsubsection*{Нефункциональные требования}

\begin{itemize}
    \item Работоспособность на стандартных рабочих станциях без выделенного GPU (инференс всех моделей на CPU). При наличии GPU приложение должно автоматически задействовать его для ускорения без дополнительной настройки.
    \item Время отклика модели сегментации не более 3~с на одно изображение в интерактивном режиме.
    \item Поддержка форматов JPEG, PNG, TIFF.
    \item Воспроизводимость результатов: одни и те же входные данные и подсказки дают одинаковый вывод.
    \item Единый формат хранения масок (PNG) и измерений (CSV / XLSX) для последующей статистической обработки.
    \item Расширяемость автоматического режима: добавление поддержки нового типа листа сводится к обучению классификатора отбора на соответствующей размеченной выборке без изменений в архитектуре приложения.
\end{itemize}

\subsection{Архитектура приложения}
\label{sec:architecture}

Приложение реализовано на языке Python с использованием фреймворка PyQt6 для графического
интерфейса и библиотеки PyTorch для инференса MobileSAM. Исходный код опубликован по адресу
\url{https://github.com/pavelrubanov/segmentation_system}.

Кодовая база разделена на три независимых слоя (рис.~\ref{fig:architecture}):

\begin{itemize}
    \item \textbf{Ядро (\texttt{src/core/}).} Вычислительные модули, не зависящие от UI:
          обёртка над MobileSAM (\texttt{predictor.py}), конвейер предобработки маски
          и канонизации листа (\texttt{leaf\_pipeline.py}), модуль измерений
          (\texttt{leaf\_measure.py}), универсальный конвейер пакетной обработки
          (\texttt{measurement\_pipeline.py}), источники фрагментов (\texttt{sources.py}),
          экспорт CSV/XLSX (\texttt{export.py}), модель данных изображения
          (\texttt{image\_data.py}) и конвенция имён файлов (\texttt{file\_naming.py}).
    \item \textbf{Интерфейс (\texttt{src/ui/}).} Окна и элементы интерфейса PyQt6: навигационное окно
          (\texttt{navigation\_window.py}), окно интерактивной сегментации
          (\texttt{segmentation/}), общие компоненты пакетных диалогов (\texttt{runner.py}),
          точки входа в пакетные режимы (\texttt{measure\_crops.py},
          \texttt{measure\_images.py}) и тема оформления (\texttt{style.py}).
    \item \textbf{Классификатор (\texttt{classifier/}).} Намеренно изолированный пакет
          с собственными зависимостями: извлечение признаков замороженной MobileNetV3-Small
          (\texttt{mobilenet.py}), детерминированная аугментация (\texttt{variants.py}),
          обучение XGBoost (\texttt{train.py}) и инференс (\texttt{predict.py}).
\end{itemize}

Точка входа~--- \texttt{src/main.py}. Скрипт принимает аргумент \texttt{-{}-checkpoint}
с путём к весам MobileSAM, создаёт единственный экземпляр \texttt{Predictor} (синглтон,

(\texttt{NavigationWindow}, рис.~\ref{fig:main_menu}). Из главного окна доступны
три режима: интерактивная сегментация (\texttt{SegmentationWindow}), пакетные
измерения по готовым фрагментам (\texttt{measure\_crops}) и полностью автоматический
режим (\texttt{measure\_images}).

\textbf{Конвенция имён файлов.} Все три источника артефактов~--- ручная сегментация,
авто-сегментация и пользовательский ввод~--- порождают и потребляют файлы по единому
шаблону: \texttt{\{stem\}\_leaf\_crop\_\{idx:04d\}.png} для фрагмента,
\texttt{\{stem\}\_mask\_\{idx:04d\}.png} для маски и
\texttt{\{crop\_stem\}\_vis.png} для визуализации измерений. Реализация и Qt-фильтры
сосредоточены в модуле \texttt{file\_naming.py}, что позволяет смешивать в одной
папке результаты разных источников и переиспользовать их между режимами без отдельной
конвертации.

\textbf{Универсальный конвейер пакетной обработки.} Оба пакетных режима
(\texttt{measure\_crops} и \texttt{measure\_images}) используют один и тот же
исполнитель \texttt{run\_pipeline} из \texttt{measurement\_pipeline.py}. Различие
сводится к источнику фрагментов~--- объекту, удовлетворяющему типу
\texttt{Callable[[str], Iterable[CropItem]]}. Для режима замеров по готовым фрагментам
источником выступает функция \texttt{disk\_source}, читающая один уже препроцессированный
PNG; для авто-режима~--- класс \texttt{AutoSegmentSource}, для каждого исходного
изображения вызывающий полный конвейер сегментации, фильтрующий результаты через
классификатор и сохраняющий промежуточные артефакты на диск. Добавление нового
сценария пакетной обработки требует только реализации очередного источника
фрагментов: фоновый исполнитель, диалог настроек, индикатор прогресса и экспорт переиспользуются.

\subsection{Конвейер обработки одного листа}
\label{sec:pipeline}

Прежде чем измерить лист, его необходимо привести к каноническому виду. Сегментирующая
модель возвращает бинарную маску, ориентированную произвольно на исходном изображении;
после ручной коррекции маска может содержать мелкие артефакты (отдельные пиксели,
дыры). Канонический вид~--- это RGB-фрагмент с обнулённым фоном, повёрнутый так, чтобы
главная ось листа располагалась вертикально, а апекс находился сверху. Только в этой
системе координат имеет смысл говорить о ``длине'' и ``поперечной ширине'' (рис.~\ref{fig:pipeline_stages}).

Конвейер реализован в модуле \texttt{leaf\_pipeline.py} и состоит из трёх шагов.

\textbf{Шаг 1: очистка маски (\texttt{preprocess\_mask}).} В произвольной маске,
полученной от MobileSAM или после ручного редактирования, могут оставаться побочные
объекты и внутренние дыры. Реализация: поиск всех внешних контуров через
\texttt{cv2.findContours}, выбор контура максимальной площади и заполнение
получившейся области через \texttt{cv2.drawContours} с флагом \texttt{FILLED}.
На выходе~--- маска \texttt{uint8} (0 либо 255), содержащая ровно одну односвязную
компоненту без дыр.

\textbf{Шаг 2: канонизация ориентации (\texttt{pca\_crop}).} По координатам активных
пикселей маски строится матрица ковариации $2 \times 2$, её собственные векторы
вычисляются через \texttt{np.linalg.eigh}. Старший собственный вектор задаёт главную
ось листа. Изображение с обнулённым фоном поворачивается аффинным преобразованием
так, чтобы главная ось совпадала с вертикалью. После поворота определяется направление
апекса: если центр масс пикселей лежит выше середины bbox, изображение переворачивается
по вертикали~--- так апекс всегда оказывается сверху. Финальный шаг~--- обрезка по
плотному bounding box активных пикселей; выходом является RGB-фрагмент, в котором
лист занимает почти всю площадь и ориентирован вертикально.

\textbf{Шаг 3: дедупликация набора масок (\texttt{dedupe\_and\_transform\_leaf}).}
В автоматическом режиме MobileSAM AMG~генерирует десятки кандидатов, среди которых
встречаются дубликаты и вложенные маски. Маски сортируются по убыванию площади;
для каждой следующей маски проверяется IoU с уже принятыми. Маска отбрасывается, если
её IoU с любой принятой превышает порог $0{,}9$. Для ускорения IoU вычисляется
не на полном изображении: предварительно для каждой маски считается её
ограничивающий прямоугольник, после чего пересечение масок выполняется только в
области пересечения этих bbox, а объединение~--- по формуле
$|A \cup B| = |A| + |B| - |A \cap B|$. К выжившим маскам применяется
\texttt{pca\_crop}; функция возвращает список пар (очищенная маска, канонический
RGB-фрагмент).

\subsection{Алгоритм параметрических измерений}
\label{sec:measure}

Модуль \texttt{leaf\_measure.py} принимает на вход \emph{канонический} RGB-фрагмент
после \texttt{pca\_crop} (главная ось вертикальна, апекс сверху, фон чёрный). Эта
конвенция позволяет отказаться от повторного построения контура и анализа главных
компонент: все измерения сводятся к работе с маской активных пикселей в системе
координат самого фрагмента (рис.~\ref{fig:measurement}).

\begin{figure}[H]
    \centering
    \includegraphics[width=0.22\linewidth]{figures/metrics_visualization.png}
    \caption{Визуализация измерений одного листа: длина и семь поперечных сечений}
    \label{fig:measurement}
\end{figure}

\textbf{Длина и максимальная ширина.} Длина листа определяется как высота
прямоугольника, ограничивающего активные пиксели фрагмента: $L = (y_{\max} - y_{\min} + 1) \cdot s$,
где $s$~--- масштабный коэффициент (мм/пиксель), равный единице, если пользователь
работает в пикселях. Максимальная ширина~--- ширина того же прямоугольника:
$W = (x_{\max} - x_{\min} + 1) \cdot s$.

\textbf{Поперечные сечения.} Для каждой доли $f \in (0, 1)$ берётся строка
$y = y_{\min} + \lfloor f \cdot (y_{\max} - y_{\min}) \rfloor$. В этой строке находятся
крайний левый и крайний правый ненулевые пиксели $x_l$ и $x_r$; ширина в сечении
равна $w_f = (x_r - x_l) \cdot s$. По умолчанию используются семь сечений с долями
$0{,}125;\; 0{,}25;\; 0{,}375;\; 0{,}5;\; 0{,}625;\; 0{,}75;\; 0{,}875$, что соответствует
равномерному разбиению длины листа на восемь интервалов; число сечений настраивается
в диалоге пакетной обработки.

\textbf{Результат.} Метод возвращает структуру \texttt{LeafMetrics} с полями
\texttt{length}, \texttt{width} и словарём \texttt{widths: dict[float, SectionWidth]}~---
для каждой доли хранятся сама ширина и координаты крайних точек, что используется
при построении визуализации. Для каждого обработанного фрагмента дополнительно сохраняется
изображение \texttt{*\_vis.png} с нанесённой осью длины и поперечными сечениями.
В режиме экспорта в XLSX миниатюра этой визуализации встраивается в ячейку таблицы,
что позволяет биологу визуально оценить корректность измерений непосредственно в
Excel, не открывая файлы по отдельности.

\subsection{Интерактивный режим сегментации}
\label{sec:interactive}

Интерактивный режим (окно \texttt{SegmentationWindow}) реализует основной сценарий
работы биолога: эксперт самостоятельно выбирает листья на изображении, а система
строит маску по подсказке и при необходимости даёт инструменты ручной коррекции
(рис.~\ref{fig:segmentation_window}). Типовой рабочий цикл показан на
рис.~\ref{fig:activity}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\linewidth]{figures/segmentation_canvas.png}
    \caption{Окно интерактивной сегментации}
    \label{fig:segmentation_window}
\end{figure}

\textbf{Холст и режимы работы.} Центральный элемент интерфейса~--- \texttt{SegmentationCanvas}
на основе \texttt{QGraphicsView} с поддержкой зума к курсору и панорамирования
(\texttt{Ctrl + ЛКМ}). Холст имеет два режима. В режиме \emph{подсказок} клики
левой и правой кнопкой мыши задают позитивные и негативные опорные точки, а
перетаскивание ЛКМ~--- ограничивающий прямоугольник; после каждого изменения
автоматически вызывается \texttt{Predictor.predict}, и над изображением отображается
полупрозрачная маска. В режиме \emph{редактирования} ЛКМ работает кистью, а ПКМ~---
ластиком; диаметр кисти регулируется ползунком в панели инструментов в диапазоне $1\ldots200$~пикселей.
Маска хранится в виде ARGB-слоя (\texttt{QImage Format\_ARGB32}); переключение между
режимами не теряет промежуточный результат.

\textbf{Буфер масок и асинхронная запись на диск.} Подтверждённые маски попадают
в список \texttt{MasksBuffer} (левая панель). Для каждой маски сохраняются
два файла~--- собственно бинарная маска и RGB-фрагмент, прошедший канонизацию по
\texttt{transform\_leaf}; при выходе из окна они копируются в выбранную пользователем
директорию. Запись PNG-файлов выполняется в пуле фоновых потоков
(\texttt{ThreadPoolExecutor} с четырьмя потоками), а в основном потоке хранятся
только пути и уменьшенная копия исходного изображения (longest side $384$~пикселя)
для отрисовки иконок $128 \times 128$. Это позволяет работать с 4K-кадрами, не
блокируя UI на запись больших PNG.

\textbf{Кнопка авто-сегментации.} В панели инструментов доступна кнопка автоматической
сегментации текущего изображения~--- она запускает функцию \texttt{auto\_segment}
(см.\ раздел~\ref{sec:auto_mode}) в отдельном \texttt{QThread}, после чего

\textbf{Замеры по готовым фрагментам (\texttt{measure\_crops}).} Сценарий используется,

сразу подаётся в \texttt{measure\_leaf}. По умолчанию формат экспорта~--- CSV,

выполняются без участия пользователя. Источник фрагментов~--- \texttt{AutoSegmentSource}:
для каждого изображения запускается полный конвейер автоматической сегментации
(раздел~\ref{sec:auto_mode}), полученные фрагменты пропускаются через классификатор
отбора (раздел~\ref{sec:classifier}), оставляются только классифицированные как
\texttt{good\_leaf}, после чего маски и фрагменты сохраняются в подпапки
\texttt{masks/} и \texttt{crops/} выходной директории. Дополнительно в диалоге настроек присутствует выпадающий список типа
листа~--- из него выбирается обученная модель отбора (\texttt{models/model\_<type>.pkl}).

миниатюры визуализаций измерений; такой отчёт позволяет биологу пройти по списку

\subsection{Полностью автоматический режим обработки}
\label{sec:auto_mode}

Полностью автоматическая сегментация реализована единой функцией
(\texttt{auto\_segment} в модуле \texttt{leaf\_pipeline.py}), вызываемой
как из интерактивного окна (по кнопке авто-сегментации), так и из пакетного
режима. На вход подаётся RGB-изображение и инициализированный экземпляр
\texttt{Predictor}, на выход~--- список пар (бинарная маска, канонический
RGB-фрагмент) в координатах исходного изображения. Внутри объединяются два
содержательных шага.

\begin{enumerate}
    \item \textbf{Автоматическая сегментация.} На исходном изображении
          запускается \texttt{SamAutomaticMaskGenerator} (метод
          \texttt{predict\_all} обёртки \texttt{Predictor}) с параметрами
          \texttt{points\_per\_side}$\,= 8$, \texttt{points\_per\_batch}$\,= 64$,
          \texttt{pred\_iou\_thresh}$\,= 0{,}80$, \texttt{stability\_score\_thresh}$\,= 0{,}88$.
          Маски, площадь которых меньше $0{,}1\,\%$ или больше $70\,\%$ кадра,
          отфильтровываются.
    \item \textbf{Очистка и дедупликация.} К полученным маскам-кандидатам
          применяется конвейер очистки маски и канонизации листа,
          описанный в разделе~\ref{sec:pipeline}: дубликаты отбрасываются по
          IoU, оставшиеся маски очищаются и приводятся к каноническому виду
          (главная ось вертикальна, апекс сверху, фон обнулён).
\end{enumerate}

Прямолинейная реализация описанных двух шагов на 4K-кадре оказывается слишком
медленной для лабораторного цикла; способы её ускорения и архитектурные
изменения, позволяющие достигнуть приемлемого времени работы, рассмотрены
в подразделе~\ref{sec:auto_perf}.

Результат работы автоматической сегментации, наложенный на исходное изображение,
показан на рис.~\ref{fig:autosegment_result}: каждая найденная маска отрисована
своим цветом, видны как корректные одиночные листья, так и кандидаты с дефектами,
которые далее отсеиваются классификатором.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\linewidth]{figures/segmentation_canvas_autosegment_result.png}
    \caption{Результат автоматической сегментации в окне ручной разметки}
    \label{fig:autosegment_result}
\end{figure}

В отличие от интерактивного режима, где эксперт верифицирует каждую маску
визуально, в полностью автоматическом режиме ошибка сегментации автоматически
попадает в выходную таблицу и влияет на результат исследования. Поэтому за
этапом автоматической сегментации обязательно следует фильтрация по классификатору
отбора (раздел~\ref{sec:classifier}): только маски, классифицированные как
корректный одиночный лист, попадают в итоговый набор.

\subsection{Отбор масок: классификатор листьев}
\label{sec:classifier}

В полностью автоматическом режиме каждая маска-кандидат, полученная на этапе
автоматической сегментации, проходит через классификатор отбора: только объекты,
классифицированные как корректные одиночные листья, попадают в итоговую таблицу
измерений. Подзадача оформлена в виде независимого пакета \texttt{classifier/}
и решается двухэтапной моделью~--- замороженный экстрактор признаков на основе
свёрточной сети, предобученной на ImageNet, и градиентный бустинг XGBoost,
обучаемый на этих признаках (рис.~\ref{fig:classifier_pipeline}). Такой
подход не требует дообучения свёрточной части под предметную область и
устойчиво работает на сотнях обучающих примеров на тип листа.

\subsubsection*{Извлечение признаков}

Используется MobileNetV3-Small из \texttt{torchvision}, предобученная на ImageNet.
Финальный классификационный слой заменяется на \texttt{Identity}, веса сети
заморожены. Каждый фрагмент приводится к квадрату $512 \times 512$ пикселей с
сохранением пропорций (letterbox~--- паддинг чёрным), нормализуется по
статистикам ImageNet и пропускается через сеть; на выходе~--- вектор $576$
признаков. Такой подход~--- использование сети, обученной на ImageNet, в режиме
переноса знаний (transfer learning)~--- обоснован тем, что нижние и средние
слои MobileNetV3 уже умеют выделять текстуру, цвет и базовые формы; этого
достаточно для последующего разделения трёх грубых классов (хороший лист~/
плохой лист~/ не лист), и нет необходимости обучать свёрточную часть на
небольшой размеченной выборке листьев. Извлечение признаков выполняется один
раз на этапе сборки обучающего датасета (модуль \texttt{mobilenet.py},
CLI-команда \texttt{python -m classifier.mobilenet}); в инференсе те же признаки
вычисляются на лету.

\subsubsection*{Аугментация}

Размеченная выборка для каждого типа листа сравнительно небольшая (от
$\sim 100$ до $\sim 600$ примеров класса \texttt{good\_leaf}), поэтому для
каждого исходного фрагмента генерируются шесть детерминированных вариантов:
горизонтальное отражение (два состояния) перемножается с тремя цветовыми
трансформациями~--- оригинал, осветление ($\text{brightness}=1{,}2$,
$\text{contrast}=1{,}15$, $\text{saturation}=0{,}9$) и затемнение ($0{,}8$,
$0{,}85$, $1{,}1$). Аугментация фиксирована в коде (модуль \texttt{variants.py})
и одинакова между запусками, что обеспечивает воспроизводимость экспериментов.
Признаки MobileNetV3 вычисляются для каждого варианта отдельно и сохраняются в
обучающий кэш \texttt{features.npz}, что устраняет необходимость пересчёта при
повторных обучениях XGBoost.

\subsubsection*{Классификатор}

XGBoost (\texttt{XGBClassifier}, \texttt{multi:softprob}) с гиперпараметрами:
$300$ деревьев, максимальная глубина $6$, темп обучения $0{,}1$,
\texttt{subsample}$\,= 0{,}8$, \texttt{colsample\_bytree}$\,= 0{,}8$,
\texttt{tree\_method}$\,=$ \texttt{"hist"}. Веса классов обратно пропорциональны
их численностям~--- это компенсирует существенный дисбаланс
($\sim 100\!-\!600$ позитивных против $1500$ \texttt{bad\_leaf} и $\sim 1000$
\texttt{non\_leaf}). Модель обучается на трёх классах:

\begin{itemize}
    \item \texttt{good\_leaf}~--- корректный одиночный лист, пригодный для
          морфометрических измерений;
    \item \texttt{bad\_leaf}~--- лист с дефектами (перекрытия с другими
          объектами, неполная форма, рваные края);
    \item \texttt{non\_leaf}~--- объект нелистовой природы (фон, тень, артефакт
          сегментации).
\end{itemize}

На этапе фильтрации в \texttt{AutoSegmentSource} сохраняются только фрагменты класса
\texttt{good\_leaf}; разделение между \texttt{bad\_leaf} и \texttt{non\_leaf}
имеет диагностическое значение (позволяет в будущем по-разному обрабатывать
эти два типа отказов), но в текущей реализации они одинаково отбрасываются.

\subsubsection*{Поддержка нескольких типов листьев \textit{Sphagnum}}

Морфология листьев у разных групп существенно различается~--- ветвевые листья
имеют гладкий контур, стеблевые \textit{capillifolium} отличаются вытянутой
формой, \textit{girgensohnii}~--- более широкой пластинкой. Один универсальный
классификатор работает существенно хуже специализированных, поэтому для каждого
типа листа обучается отдельная модель \texttt{models/model\_<type>.pkl}: в качестве
позитивного класса берутся ``хорошие'' листья из соответствующей подпапки
\texttt{good\_leaf\_<type>}, негативные классы \texttt{bad\_leaf} и
\texttt{non\_leaf} являются общими для всех типов и не требуют повторной
разметки. В диалоге авто-режима пользователь выбирает тип листа~--- система
автоматически подгружает соответствующие веса. Добавление поддержки нового типа
сводится к разметке выборки \texttt{good\_leaf\_<type>} и запуску команды
\texttt{python -m classifier.train}: код приложения при этом не меняется.

\subsubsection*{Процедура обучения}

Использовалось hold-out разбиение $60/20/20$ (train/val/test) с фиксированным
\texttt{random\_state}: train размножается шестикратно через аугментацию,
val и test остаются без аугментации (только базовый вариант). На train
обучается XGBoost с указанными гиперпараметрами; val используется для
финального контроля (без раннего останова, поскольку число деревьев фиксировано),
test~--- для независимой оценки качества. Дополнительно проведена $5$-кратная
стратифицированная кросс-валидация на полном датасете для оценки устойчивости
качества к выбору разбиения.

\subsubsection*{Результаты}

Распределение классов в обучающих выборках существенно несбалансированное:
позитивный класс \texttt{good\_leaf} составляет от $4{,}6\,\%$ (для
\texttt{capillifolium}) до $18{,}5\,\%$ (для \texttt{branch}) объёма выборки
(табл.~\ref{tab:clf_balance}). Поэтому \emph{общая accuracy не является
информативной метрикой}~--- тривиальный классификатор, всегда предсказывающий
\texttt{bad\_leaf}, получит accuracy около $0{,}55$, не приняв при этом ни
одного корректного листа. В качестве сводной оценки используется macro-F1~---
невзвешенное среднее F1-меры по трём классам, дающее равный вес каждому из них
независимо от поддержки. Все метрики приводятся в формате
$\text{mean} \pm \text{std}$, где среднее и выборочное стандартное отклонение
вычислены по пяти фолдам стратифицированной кросс-валидации.

\subsection{Оптимизация производительности автоматической сегментации}
\label{sec:auto_perf}

Прямолинейная реализация автоматической сегментации~--- запуск AMG на исходном
4K-кадре, очистка масок и сохранение PNG-файлов синхронно с UI~--- работала
недопустимо медленно. На ноутбуке без GPU обработка одного кадра
$3840 \times 2160$ занимала $\approx 18{,}7$~с, что для типового пакета в
50~снимков давало порядка пятнадцати минут. В интерактивном режиме UI зависал
на $\approx 3$~с при сохранении PNG-файлов. Это сделало необходимым
целенаправленную оптимизацию конвейера.

Замеры выполнялись на ноутбуке (CPU, без GPU) на 4K-кадре с 13 листьями
\textit{Sphagnum} после дедупликации. Архитектурные шаги оптимизации и их

Шаги обсуждаются в порядке применения.

\textbf{Шаг 1. Уменьшение масштаба перед AMG.} Кодировщик SAM принудительно
приводит вход к квадрату со стороной $1024$~пикселей, поэтому стоимость
кодировщика от исходного разрешения не зависит. Однако пост-обработка масок
(NMS, фильтрация, \texttt{stability\_score}) масштабируется как $O(W \cdot H \cdot N)$
и для 4K оказывается в четыре раза дороже, чем для HD. После уменьшения входа
в два раза время AMG сократилось с $6{,}30$~с до $0{,}50$~с. Маски затем
масштабируются обратно к исходному разрешению через \texttt{INTER\_NEAREST};
потеря $\sim 2$~пикселей точности на границе не влияет на последующие измерения,
поскольку дальнейшая обработка работает с целыми областями маски.

\textbf{Шаг 2. Единая точка входа автоматической сегментации.} После шага~1
конвейер выглядел как метод \texttt{predict\_all} обёртки \texttt{Predictor}
(внутри~--- уменьшение и масштабирование к исходному разрешению), за которым
следовали дедупликация и канонизация на полном разрешении. Дедупликация
выполнялась на масках, формально 4K, но фактически содержащих информацию
HD-разрешения~--- то есть тяжёлые попарные операции IoU и обработка контуров
делались на $8{,}3$~Mpx без необходимости. Решение~--- вынести шаг уменьшения
масштаба из обёртки \texttt{Predictor} в новую функцию (\texttt{auto\_segment}
в модуле \texttt{leaf\_pipeline.py}), передавать на дедупликацию именно
HD-маски, а масштабирование к полному разрешению выполнять отдельным финальным
циклом. Время этапа дедупликации сократилось с $4{,}99$~с до $1{,}17$~с.

\textbf{Шаг 3. Копирование по bbox вместо \texttt{image.copy()} в цикле
масштабирования.} В финальном цикле для каждой маски выполнялось
\texttt{masked = image.copy()}~--- полная копия 4K-кадра ($\approx 25$~МБ),
после чего вне маски пиксели обнулялись. На тринадцати листьях это давало
$\approx 325$~МБ копий и тяжёлую \texttt{warpAffine} на полном кадре. Замена:
считать bbox каждой маски и копировать только соответствующий регион
(\texttt{numpy}-срез возвращает представление, копирование выполняется только
при \texttt{.copy()}). Время цикла масштабирования: $2{,}89 \to 0{,}69$~с.

\textbf{Шаг 4. Пропуск повторной PCA-канонизации.} Дедупликация уже выполняет
PCA-канонизацию каждого выжившего листа на HD-кадре (внутри своего цикла);
в финальном цикле масштабирования этот результат игнорировался, и канонизация
запускалась повторно для полного разрешения. Один и тот же лист
обрабатывался дважды. Замена: переиспользовать HD-фрагмент и масштабировать
его к эквиваленту полного разрешения билинейной интерполяцией. Размытие на
$2\!-\!3$~пикселя при таком масштабировании некритично~--- классификатор всё
равно приводит фрагмент к $512 \times 512$, а измерения работают с
целочисленными координатами bbox. Время цикла масштабирования:
$0{,}69 \to 0{,}10$~с.

\textbf{Шаг 5. Копирование по bbox внутри дедупликации.} Тот же приём, что и
в шаге~3, но применённый к финальному циклу внутри функции дедупликации~---
там, где для каждой выжившей маски вычислялась PCA-канонизация. До замены
копировался полный HD-кадр ($\approx 6$~МБ) на каждую маску~--- $\approx 78$~МБ
копий и аффинное преобразование на полном HD. После~--- работа только в bbox
каждой маски. Время этапа дедупликации: $1{,}17 \to 0{,}57$~с.

\textbf{Шаг 6. Асинхронная запись PNG и кэш миниатюры изображения.}
В \texttt{MasksBuffer} запись каждой маски на диск выполнялась синхронно: копия
полного 4K-кадра $\to$ обнуление пикселей вне маски $\to$ \texttt{Image.save}
$\to$ загрузка PNG обратно для отрисовки миниатюры $128 \times 128$. На тринадцати
масках это давало $\approx 2{,}9$~с зависания UI. Решение состоит из трёх
компонентов: (а)~PNG-запись вынесена в \texttt{ThreadPoolExecutor} из четырёх
потоков, futures хранятся в словаре \texttt{\_pending} и при необходимости
ожидаются точечно; (б)~для отрисовки иконок один раз на изображение строится
\texttt{image\_thumb}~--- уменьшенная копия исходника с длинной стороной
$384$~пикселя ($\approx 250$~КБ вместо 25~МБ), все миниатюры $128 \times 128$
строятся из неё; (в)~операция ``обнулить вне маски'' заменена на
\texttt{cv2.bitwise\_and}~--- один SIMD-проход вместо двух. Время блока сохранения:
$2{,}9 \to 0{,}12$~с, UI больше не зависает.

\textbf{Итог.} Совокупное ускорение~--- $\approx 10{,}4$~раза ($18{,}7 \to 1{,}8$~с
на одном кадре). Для типового пакета в 50~снимков это означает сокращение времени
обработки с пятнадцати до полутора минут и переводит автоматический режим из
заведомо неприемлемого по времени в работоспособный для лабораторного цикла.

\subsection{Тестирование}
\label{sec:testing}

Корректность модуля измерений верифицирована модульными тестами:

\begin{itemize}
    \item \textit{Прямоугольная маска} заданной ширины и высоты~--- ожидаемые длина
          и максимальная ширина совпадают с расчётными значениями с погрешностью

    \item \textit{Эллиптическая маска}~--- проверяется симметричность профиля

    \item \textit{Интеграционный тест}~--- загрузка реальных тестовых изображений,
          получение маски через \texttt{Predictor}, запуск \texttt{measure\_leaf}
          и проверка формата CSV-таблицы.
\end{itemize}

Корректность конвейера автоматической сегментации и работы UI проверялась
интеграционно~--- запуском приложения на тестовом наборе изображений и визуальной
сверкой полученных масок и измерений с ручной разметкой эксперта (см.\ раздел~\ref{sec:experiment}).

\subsection{Сборка и публикация}
\label{sec:packaging}

Целевая аудитория~--- сотрудники биологической лаборатории, для которых установка
Python и его зависимостей не является штатной операцией. Доставка реализована
в виде единого автономного дистрибутива под Windows~64 на базе PyInstaller~6.11.1
в режиме \texttt{-{}-onedir} (конфигурация~--- \texttt{packaging/sphagnum.spec},
скрипт~--- \texttt{packaging/build\_windows.ps1}). Одна и та же спецификация применима
к CPU- и CUDA-вариантам \texttt{torch}.

Статический анализатор PyInstaller не видит динамических импортов
\texttt{mobile\_sam}, \texttt{timm} (модели регистрируются декораторами) и
\texttt{xgboost} с \texttt{sklearn} (загружаются при \texttt{pickle.load} весов
классификатора), поэтому эти пакеты подключаются явно через \texttt{collect\_all};
для пакета \texttt{classifier}~--- \texttt{collect\_submodules}. Тяжёлые
неиспользуемые модули (\texttt{tkinter}, \texttt{IPython}, \texttt{jupyter},
\texttt{pytest}) исключаются из дистрибутива.

Отдельно потребовался обработчик времени выполнения для PyTorch: при первом
запуске собранного приложения возникает \texttt{WinError 1114} на
\texttt{c10.dll}, поскольку встроенный загрузчик \texttt{torch} использует
строгий \texttt{LoadLibraryExW}, несовместимый с контекстом замороженного
PyInstaller-приложения. Обработчик
(\texttt{packaging/runtime\_hooks/torch\_dll\_path.py}) предзагружает критические
DLL через \texttt{ctypes.CDLL} с более мягкими флагами; собственный цикл
\texttt{torch} затем получает уже готовые дескрипторы.

Итоговый дистрибутив \texttt{dist/sphagnum-cpu-win64/} занимает $\approx 920$~МБ
и распространяется в виде RAR-архива ($\approx 327$~МБ). Запуск~--- двойным
кликом по \texttt{sphagnum.exe}; консольное окно подавлено
(\texttt{console=False}), стандартные потоки перенаправляются в \texttt{stdout.log}
и \texttt{stderr.log} в директории дистрибутива.

\textbf{Гибридный классификатор с обучаемой свёрточной ветвью.} Архитектура

(\texttt{Conv}--\texttt{BN}--\texttt{ReLU}--\texttt{MaxPool}, $3 \to 128$ каналов)
обрабатывала RGB-фрагмент $96 \times 96$, параллельный MLP принимал
21-мерный вектор ручных признаков (статистики цвета RGB и HSV, инварианты Ху,
эксцентриситет, плотность заполнения, solidity, circularity); выходные
представления конкатенировались и подавались в общую классификационную голову
$192 \to 64 \to 3$. Полная модель достигла точности $0{,}981$ на test
($156$ примеров), отключение CNN-ветви снижало точность до $0{,}853$~--- то есть
свёрточная часть действительно вносила вклад. Однако подход имеет два недостатка:
во-первых, требует собственного обучения свёрточной части и расширения
обучающей выборки при добавлении нового типа листа; во-вторых, ручные признаки
требуют согласованности с конвейером препроцессинга, что усложняет сопровождение.
Схема \mbox{MobileNetV3 + XGBoost} даёт сопоставимое качество без необходимости
обучать свёрточную часть~--- замороженный экстрактор, обученный на миллионах
изображений ImageNet, не уступает специально обучаемой лёгкой CNN на сотнях
примеров.

\textbf{Одноклассовое обнаружение аномалий.} Рассматривались три модели,
обучаемые исключительно на классе \texttt{good\_leaf} и выдающие числовой score
аномальности: свёрточный автоэнкодер с per-pixel MSE-оценкой, гауссова смесь
(GMM) на 21-мерных ручных признаках и \texttt{OneClassSVM} с RBF-ядром на тех
же признаках. GMM и OC-SVM показали высокое качество (AUC $\approx 0{,}99$,
F1 $\approx 0{,}98$ на бинарной задаче), автоэнкодер~--- заметно ниже
(AUC $0{,}740$): ошибка реконструкции слабо разделяет хорошие листья и аномалии
при работе с RGB-фрагментами на чёрном фоне. Несмотря на конкурентные метрики,
подход отбракован по двум причинам. Во-первых, для подбора порога отсечения
всё равно требуется размеченная аномальная выборка~--- формальное преимущество
``обучение только на хороших примерах'' на практике не реализуется.
Во-вторых, схема не различает \texttt{bad\_leaf} и \texttt{non\_leaf}, тогда
как разделение этих типов отказов сохраняет диагностическую ценность для
последующего анализа ошибок. По совокупности факторов схема с обучением с учителем и
тремя классами оказалась более удобной как с точки зрения интеграции в
приложение, так и с точки зрения интерпретации результатов.
